# 실제 API 환경에서의 분산 LLM 에이전트 아키텍처에 대한 실증적 분석

## 요약 (Abstract)
기존의 분산 에이전트 아키텍처 시뮬레이션은 종종 인위적인 모의(Mock) 지연 시간에 의존해 왔습니다. 본 연구에서는 실제 LLM API(Gemini 3.1 Flash-Lite)와 학술 데이터셋(CNN-Dailymail 및 HotpotQA)을 사용하여 대표적인 세 가지 다중 에이전트 오케스트레이션 토폴로지(Master-Worker, Queue-Based, Swarm)를 벤치마킹합니다. 실증적 결과에 따르면, 실제 네트워크 I/O 지연 시간이 반영될 경우 기존 모의 환경에서 관찰되었던 성능 패러다임이 완전히 뒤바뀌는 것을 확인할 수 있었습니다. 특히, 에이전트 간 직접 통신에 최적화되었다고 여겨지는 Swarm 구조는 API 대기 시간(Blocking)으로 인해 심각한 성능 저하를 겪었습니다. 반면, 다중 홉 순차 추론(Multi-hop QA)에서는 Queue-Based 아키텍처가 월등한 처리량과 결함 허용(Fault Tolerance) 능력을 입증했습니다.

---

## 1. 서론
LLM 애플리케이션의 복잡성이 증가함에 따라, 작업 부하(Workload)를 다수의 자율 에이전트에 분산시키는 방식이 주목받고 있습니다. 그러나 이러한 에이전트들을 조율(Orchestration)하는 방식에 따라 통신 오버헤드와 병렬 처리 효율성 사이에 분명한 트레이드오프가 발생합니다.

본 논문에서는 로컬 모의 추론 방식을 넘어서, 예측 불가능한 네트워크 지연 시간과 트래픽 제한(HTTP 429)이 존재하는 **실제 API 호출** 환경으로 벤치마크를 고도화했습니다. 더 나아가, 맵리듀스(Map-Reduce) 기반 요약을 위한 **CNN-Dailymail**과 연쇄 추론(Chain-of-Thought)을 위한 **HotpotQA** 등 검증된 학술 데이터셋을 도입하였습니다.

---

## 2. 연구 방법론
### 2.1 평가 대상 아키텍처
1. **Master-Worker (중앙 집중형)**: 마스터 노드가 하위 작업을 분배하고 결과를 병합합니다.
2. **Queue-Based (상태 분산형)**: 워커 노드들이 공유 메시지 큐(Broker)로부터 비동기적으로 작업을 가져와 처리합니다.
3. **Swarm (P2P 네트워크)**: 중앙 통제자 없이 에이전트들이 서로 직접 통신하며 협력합니다.

### 2.2 워크로드 및 데이터셋
- **Task A (맵리듀스)**: CNN-Dailymail 테스트 세트에서 무작위로 기사를 추출하여 $N$개의 청크로 분할합니다. 각 에이전트가 독립적으로 청크를 요약한 뒤, 마지막 병합 단계를 거쳐 최종 요약본을 완성합니다.
- **Task B (멀티홉 QA)**: 에이전트가 HotpotQA의 문제를 해결하기 위해 필요한 중간 질문을 동적으로 생성하고, 발견된 단서(Context)를 파이프라인의 다음 에이전트에게 순차적으로 전달합니다.

### 2.3 복원력 메커니즘 (Resilience)
실제 API의 불안정성에 대응하기 위해, 지수적 백오프(`tenacity`)와 결합된 자동 **API 키 순환(Key Rotation)** 전략을 구현하여 트래픽 제한(Rate Limit) 고갈을 투명하게 완화했습니다. 또한 시스템의 자동 복구(Auto-healing) 능력을 평가하기 위해 노드 장애(Crash) 주입 테스트를 실시했습니다.

## 3. 실험 결과

### 3.1 지연 시간(Latency) 분석

| 아키텍처 (워커 수=4) | Task A (P50 지연 시간) | Task B (P50 지연 시간) |
| :--- | :--- | :--- |
| **Master-Worker** | **1.55초** | 4.99초 |
| **Queue-Based** | 3.23초 | **2.61초** |
| **Swarm** | 5.83초 | 6.23초 |

*(참고: 여러 반복(Iteration)에 걸쳐 10개의 구성을 테스트하여 집계한 데이터입니다.)*

![Latency Comparison](latency.png)

**관찰 1 (Task A):** 병렬성이 높은 맵리듀스 작업에서는 **Master-Worker** 아키텍처가 가장 효율적(1.55초)이었습니다. 중앙에서 위임하므로 API 준비 시간이 최소화되어, 독립적인 요약 작업을 빠르게 분산(Fan-out)시킬 수 있었습니다.

**관찰 2 (Task B):** 연쇄 추론(Task B)에서는 **Queue-Based** 아키텍처가 압도적인 성능(2.61초)을 보였습니다. 멀티홉 QA는 단서(Context)를 순차적으로 넘겨야 하는데, 비동기 메시지 큐 방식은 워커들이 대기 상태(Blocking)에 빠지는 것을 막고 다음 추론 단계를 즉각적으로 유휴 워커에게 배급하기 때문입니다.

**관찰 3 (Swarm의 병목 현상):** 기존 모의(Mock) 시뮬레이션에서 Swarm 구조는 종종 높은 처리량을 보였습니다. 그러나 실제 API 환경(요청당 1~3초 소요)에서는 P2P 동기화 로직으로 인해 극심한 대기 오버헤드(5.83~6.23초)가 발생했습니다. 동료 에이전트의 응답을 기다리는 시간이 전체 네트워크의 병목으로 작용했습니다.

### 3.2 처리량(Throughput) 분석

![Throughput Comparison](throughput.png)

처리량 지표 역시 지연 시간 분석과 완벽히 일치합니다. Master-Worker는 Task A에서 초당 약 1회의 요청을 처리한 반면, Swarm은 동일한 4-워커 제한 아래에서 초당 약 0.3회로 크게 저하되었습니다.

### 3.3 결함 허용 및 자동 복구 (Fault Tolerance & Auto-Healing)

시스템의 견고성을 테스트하기 위해 **Queue-Based (Task A, W=4)** 구성에 무작위 워커 노드 장애(Crash)를 주입했습니다:

* **정상 상태 P50 지연 시간**: 3.23초
* **장애 주입 시 P50 지연 시간**: 4.54초

추론 도중 활성 워커 하나가 갑자기 죽거나 429 에러가 발생하는 상황에서도, Queue-Based 아키텍처는 유실된 작업을 감지하여 큐에 재등록(Re-queue)하고 살아남은 워커들을 동원해 작업을 완수했습니다. 정상 대비 약 1.3초의 페널티는 API 키 로테이션 대기 시간과 재실행 오버헤드가 포함된 수치로, 교착 상태(Deadlock)를 완벽히 방지하며 프로덕션 수준의 복원력을 증명했습니다.

---

## 4. 결론
실제 환경에서 LLM 에이전트를 배포할 때는 높은 네트워크 I/O 지연과 API의 불안정성을 반드시 고려해야 합니다. 본 벤치마크의 결론은 다음과 같습니다:
1. **Master-Worker** 아키텍처는 순수 병렬 작업(예: 맵리듀스)에 투입되어야 합니다.
2. **Queue-Based** 아키텍처는 비동기적인 특성과 본질적인 결함 허용 능력 덕분에 다단계 추론 파이프라인(예: Chain-of-Thought)에서 가장 우수합니다.
3. **Swarm (P2P)** 토폴로지는 이론적으로 탈중앙화되어 있으나, 외부 API 추론 시간에 의존하게 될 경우 심각한 연쇄적 지연(Cascading delay) 병목을 겪게 됩니다.
