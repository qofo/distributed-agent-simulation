# LLM 분산 에이전트 시스템에서 Mock 벤치마크의 환상과 API I/O 병목의 현실: 아키텍처별 성능 역전 현상 분석

## 초록 (Abstract)
최근 LLM 기반 멀티 에이전트 시스템이 복잡한 추론과 파이프라인 처리를 위해 도입되고 있으나, 이를 평가하기 위한 대부분의 벤치마크는 고정된 짧은 지연 시간(Mock)을 가정하고 이루어집니다. 본 연구는 Mock 벤치마크 환경(100ms 지연)과 실제 상용 LLM API 환경(2000ms 이상의 네트워크 I/O 병목 및 Rate Limit) 간에 발생하는 극명한 성능 역전 현상을 분석합니다. 실험 결과, Mock 환경에서 압도적인 우위를 보였던 Swarm(P2P) 아키텍처는 실제 환경에서 심각한 병목을 겪으며 최하위로 추락한 반면, 오버헤드가 크다고 여겨졌던 Queue-Based 시스템은 I/O 대기 시간을 훌륭하게 흡수하며 실제 환경에서 가장 높은 복원력과 처리량을 달성했습니다. 본 논문은 LLM 오케스트레이션 설계 시 CPU 연산 중심이 아닌 비동기 네트워크 I/O 중심의 설계가 필수적임을 제안합니다.

---

## 1. 서론 (Introduction)
LLM(대형 언어 모델) 에이전트를 여러 개 연결하여 복잡한 태스크를 해결하는 분산 아키텍처 연구가 활발합니다. 하지만 기존 연구들은 대부분 시뮬레이션(Mock) 기반의 짧은 고정 지연 시간 하에서 아키텍처의 이론적 효율성만을 측정하는 한계를 지닙니다.
초기 연구(Report 1, 2)에서는 Swarm 구조가 중앙 병목 없이 에이전트 간 직접 통신을 통해 가장 완벽한 성능을 낸다고 결론 내렸으나, 실제 상용 API를 연동한 후속 실험에서는 그 결론이 완전히 뒤집혔습니다. 이에 본 연구는 지연 시간을 실제 API 수준(2000ms)으로 극대화한 시뮬레이터를 통해 원인을 분석하고, "왜 아키텍처 성능이 역전되는지" 규명합니다.

---

## 2. 실험 설계의 결함과 재조정 (Revisiting Experimental Design)
기존 Mock 기반 실험에서 제기된 치명적인 결함과 분석의 한계를 수정하기 위해, 시뮬레이션 가정을 실제 환경에 맞게 재조정했습니다.

### 2.1. Mock 벤치마크의 환상 (Illusion of Mock Environments)
* **API 지연 시간의 부재**: 기존 실험은 50~100ms의 짧은 고정 지연을 사용했습니다. 이 경우 큐를 읽고 쓰는(Locking) 시간이나 중앙 마스터의 통신 오버헤드가 상대적으로 크게 부각되어 Queue-based의 성능이 평가절하되었습니다.
* **수정된 실험 (v4)**: 실제 LLM API가 생성에 소요하는 평균 시간인 `2000ms`를 기본 지연(Latency)으로 설정하여, 연산 오버헤드보다 **I/O 대기(Blocking)**가 전체 시스템을 지배하도록 재설계했습니다.

### 2.2. Swarm 구현체의 숨겨진 함정 (Hidden Cheat in P2P Mock)
기존 Mock 벤치마크에서 Swarm이 비정상적으로 좋게(Master 대비 W4에서 거의 2배 성능) 나온 이유를 코드로 계측한 결과, 놀라운 사실이 밝혀졌습니다.
* **동시성 제어 부재**: Master와 Queue 구조는 스레드 풀(Thread Pool)을 엄격히 `W(워커 수)`만큼 제한하여 큐 대기를 시뮬레이션한 반면, 기존 Swarm의 Mock 구현체는 청크 단위로 **무제한 스레드를 생성(Spawn)**하고 있었습니다.
* **Real API와의 괴리**: Mock 환경에서는 무한 스레드 생성이 CPU 컨텍스트 스위칭만으로 처리되어 엄청난 속도를 냈지만, 실제 API 환경에 붙는 순간 수십 개의 스레드가 동시에 HTTP 커넥션을 맺으려다 **API Rate Limit (HTTP 429)**에 부딪혀 시스템이 연쇄적으로 붕괴(Cascading Failure)한 것입니다. 이것이 Swarm이 Real API에서 갑자기 꼴등으로 추락한 근본 원인이었습니다.

### 2.3. 공정한 예외 처리 (Fair Error Handling)
Master Crash 상황에서 작업이 무시되고 정상 종료되어 Throughput이 올라가는 통계적 오류(Anomaly)를 수정하였습니다. Worker Crash 시 `CrashSimulationError`를 반환하도록 고쳐, Queue 기반 시스템의 DLQ(Dead Letter Queue) 리트라이 장점과 대조되도록 공정하게 비교했습니다.

## 3. 실험 결과 및 분석 (Results & Analysis)

### 3.1. I/O 지배 상황에서의 확장성 역전 (Task A)
2000ms 지연 환경(v4)에서의 Map-Reduce 태스크 결과입니다.

| Architecture | Throughput (W2) | Throughput (W4) | Throughput (W16) | Throughput (W32) |
|--------------|----------------|----------------|-----------------|-----------------|
| Master-Worker| 0.354          | 0.592          | 1.125           | 1.053           |
| Queue-Based  | 0.382          | 0.628          | 1.116           | 0.948           |

* **분석**: I/O가 2초 이상으로 길어지면, Master-Worker와 Queue-Based의 성능 차이는 사실상 무의미해집니다. 통신 오버헤드(Queue Wait: W16 기준 평균 0.34초)는 2초의 I/O 블로킹 시간 안에 완전히 숨어버립니다(Hide latency). 
* **Worker 포화**: W16에서 W32로 증가시켜도 Throughput이 늘어나지 않고 1.1 내외로 정체됩니다. 이는 큐 메시지 브로커의 락(Lock) 병목이 아니라, 태스크를 분할하고 합병하는 과정 자체의 한계(Amdahl's Law)에 도달했기 때문입니다.

### 3.2. 순차 파이프라인(Task B)에서의 무의미한 병렬화
| Architecture | Throughput (W1/W2) | P50 Latency |
|--------------|-------------------|-------------|
| Monolithic (W1)| 0.509           | 15.2초      |
| Master (W2)    | 0.521           | 16.1초      |
| Queue (W2)     | 0.501           | 15.7초      |

* **분석**: 이전 논문들은 특정 구조가 Multi-hop 추론에 적합하다고 주장했으나, 데이터를 보면 **순차 체인(Chain) 구조에서는 어떤 분산 아키텍처를 쓰든 이득이 없습니다.** 2000ms의 LLM API 호출이 동기적으로 직렬화되어 있기 때문에, 오케스트레이션 툴을 도입하는 것 자체가 불필요한 시스템 복잡도만 증가시킵니다.

### 3.3. 결함 복구와 Straggler 방어력 (Queue의 진가)
가장 흥미로운 데이터는 결함(Crash/Straggler) 상황에서 나타납니다.

* **Straggler (지연 발생)**: 특정 워커가 1000~3000ms 지연을 겪는 상황에서 Queue 기반 시스템은 Throughput 저하를 10% 이내(0.628 -> 0.571)로 방어했습니다. 이는 유휴 워커가 큐에서 다음 작업을 즉시 낚아채는(Non-blocking work stealing) 비동기성 덕분입니다.
* **Crash (노드 다운)**: Master 구조에서 워커가 다운되면 파이프라인 전체가 실패하지만, Queue 구조는 10초 Timeout 후 자동 Requeue를 통해 P50 지연시간은 증가(12.4초 -> 17.6초)하지만 **전체 시스템의 처리량(Throughput 3.71, 배치 동시 완료 효과)과 100% Task 성공률**을 보장했습니다.

## 4. 결론 (Conclusion)

본 벤치마크 연구를 통해 다음을 증명했습니다.

1. **Mock 벤치마크의 위험성**: 고정 지연 시간이나 스레드 제한이 없는 단순 Mock 테스트는 분산 프레임워크의 오버헤드를 과장하고 비효율적 통신망(Unbound P2P)을 "우수하다"고 오판하게 만듭니다.
2. **LLM 에이전트의 병목은 CPU가 아닌 네트워크 I/O다**: 따라서 가장 우수한 멀티 에이전트 아키텍처는 데이터 전달이 빠른 구조가 아니라, **"긴 API 대기 시간 동안 유휴 워커가 놀지 않게 만드는(Asynchronous Non-blocking) 구조"**입니다.
3. **Queue-Based 시스템의 재발견**: 순차 파이프라인에서는 오버헤드가 있을 수 있으나, 대규모 병렬 요청(Map-Reduce) 환경이나 크래시가 빈번한 실제 분산망에서 Queue 구조는 압도적인 신뢰성과 자동 Straggler 회피 능력을 제공합니다.

향후 LLM 멀티 에이전트 오케스트레이션 프레임워크(LangGraph, AutoGen 등)를 설계할 때는 반드시 Rate Limit과 I/O Blocking을 고려한 큐잉 이론(Queueing Theory)과 백프레셔(Backpressure) 메커니즘이 도입되어야 합니다.
